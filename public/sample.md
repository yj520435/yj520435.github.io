# 아키텍처

아파치 플링크는 바운드, 언바운드 데이터 스트림의 상태유지 연산을 위한 프레임워크이자 분산 처리 엔진입니다. 플링크는 모든 공통 클러스터 환경 내에서 실행하여 인메모리 스피드로 규모에 구애받지 않고 연산을 수행하기 위해 고안되었습니다. **플링크** 아키텍처는 다음과 같습니다.

## 언바운드, 바운드 데이터 처리

모든 종류의 데이터는 이벤트의 스트림으로 생성됩니다. 신용카드 트랜잭션, 센서 측정, 머신 로그, 웹사이트 또는 모바일 어플리케이션 상에서의 사용자 상호작용 등과 같은 데이터는 스트림으로 만들어집니다.

데이터는 언바운드 또는 바운드 스트림으로 처리될 수 있습니다.

1. **언바운드 스트림**의 경우 시작은 존재하지만 정의된 끝은 존재하지 않습니다. 즉 종료 없이 생성된 데이터를 제공합니다. 언바운드 스트림은 연속적으로 처리되어야 합니다. 예를 들어 이벤트는 들어온 즉시 핸들링되어야 합니다. 입력은 무한하고 어떤 시점에 완료되는 것이 아니기 때문에 도착할 모든 입력 데이터를 대기하는 것은 불가능합니다. 결과의 완전성에 대한 판단을 가능케 하기 위해 보통 언바운드 데이터 처리에는 이벤트가 특정한 순서로 들어와야 한다는 조건이 요구되곤 합니다.
2. **바운드 스트림**은 정의된 시작과 끝이 존재합니다. 바운드 스트림은 어떤 연산을 수행하기 전에 모든 데이터를 받고서 처리될 수도 있습니다. 유한한 데이터 셋은 언제든지 정렬될 수 있기 때문에 데이터가 순서대로 들어와야 한다는 조건도 요구되지 않습니다. 바운드 스트림을 처리하는 것은 배치 프로세싱으로 알려져 있기도 합니다.

아파치 플링크는 언바운드, 바운드 데이터 셋 처리에 뛰어납니다. 시간과 상태의 정밀한 제어를 통해 플링크의 런타임은 모든 종류의 언바운드 스트림 관련 어플리케이션을 실행합니다. 바운드 스트림은 고정된 크기의 데이터 셋을 위해 특별히 설계된 알고리즘과 자료구조에 의해 내부적으로 처리되므로 우수한 성능을 제공합니다.

## 환경의 구애를 받지 않는 애플리케이션 배포

아파치 플링크는 분산 시스템이며 어플리케이션을 실행하기 위해 계산 리소스를 필요로 합니다. 플링크는 Hadoop YARN, Apache Mesos, Kubernetes와 같은 모든 공통 클러스터 리소스 매니저와 통합됩니다만, 독립 실행형 클러스터로서 실행되도록 설정할 수도 있습니다.

플링크는 앞서 나열한 리소스 매니저들의 작동이 원활하도록 설계되었습니다. 이는 플링크가 각 리소스 매니저와 자연스러운 방식으로 상호작용할 수 있게끔 하는 '리소스 매니저에 특화된 배포 모드' 덕분입니다.

플링크 어플리케이션을 배포할 때 플링크는 어플리케이션에 설정된 병렬성을 기반으로 필요한 리소스를 자동으로 식별하고 그것들을 리소스 매니저에게 요청합니다. 실패하는 경우 플링크는 새로운 리소스를 요청하여 실패한 컨테이너를 대체합니다. 어플리케이션을 제출하거나 제어하기 위한 모든 커뮤니케이션은 REST 호출을 통해 발생합니다. 이는 다양한 환경에서 플링크의 통합이 쉽게끔 합니다.

## 규모의 구애를 받지 않는 애플리케이션 실행

플링크는 어떠한 규모의 상태유지적 스트리밍 어플리케이션도 실행할 수 있도록 설계되었습니다. 어플리케이션은 단일 클러스트 내부에서 분산되어 동시다발적으로 실행되는 수천 개의 태스크로 병렬화될 수 있습니다. 따라서 어플리케이션은 CPU, 메인 메모리, 디스크, 네트워크 IO 등을 거의 무제한으로 활용할 수 있습니다. 또한 플링크는 아주 큰 어플리케이션의 상태를 쉽게 유지할 수 있습니다. 플링크의 비동기식, 증진식 검사 알고리즘은 처리 지연에 미치는 영향을 최소화하는 동시에 정확히 한 번의 상태 일관성을 보장합니다.

사용자들은 다음과 같이 프로덕션 환경에서 실행되는 플링크 어플리케이션에 대한 인상적인 확장성 수치를 보고했습니다.

- 하루에 수조 개의 이벤트를 처리하는 애플리케이션
- 여러 테라바이트의 상태를 유지하는 애플리케이션
- 수천 개의 코어에서 실행되는 애플리케이션

## 인-메모리 성능 활용

상태유지적 플링크 어플리케이션은 로컬 상태 액세스에 최적화되어 있습니다. 태스크 상태는 항상 메모리에 유지되거나 상태의 크기가 가용한 메모리 크기를 초과하는 경우 액세스가 용이한 온디스크 데이터 구조에 유지됩니다. 따라서 태스크는 로컬(보통 메모리 내) 상태에 액세스하여 모든 연산을 수행하므로 처리 지연이 매우 낮습니다. 플링크는 장애가 발생하는 경우 주기적으로, 그리고 비동기적으로 로컬의 상태를 지속가능한 저장소에 체크포인팅하여 정확히 한 번의 상태 일관성을 보장합니다.

# 애플리케이션

아파치 플링크는 언바운드, 바운드 데이터 스트림의 상태유지적 연산을 위한 프레임워크입니다. 플링크는 각기 다른 추상 레벨의 다양한 API를 제공하고 일반적인 유스케이스에 대한 라이브러리를 제공합니다.

이 장에서는 사용이 용이한 플링크의 API와 라이브러리에 대해 알아보도록 하겠습니다.

## 스트리밍 애플리케이션을 위한 빌딩 블록

스트림 프로세싱 프레임워크에 의해 구축, 실행될 수 있는 어플리케이션의 종류는 해당 프레임워크가 스트림, 상태, 시간을 얼마나 잘 제어하는지에 따라서 정해집니다. 스트림 처리 어플리케이션을 위한 빌딩 블록을 그려보고 그것들을 핸들링하는 플링크의 접근 방식에 대해 설명해보겠습니다.

## 스트림

스트림은 스트림 프로세싱의 근본 양상이지만 스트림이 어떻게 처리되어질 수 있고 어떻게 처리되어야만 하는지에 영향을 미치는 또다른 특징을 가질 수 있습니다. 플링크는 어떠한 종류의 스트림도 핸들링할 수 있는 다목적 처리 프레임워크입니다.

- **바운드, 언바운드 스트림**: 스트림은 제한이 없거나(unbounded) 제한이 있을(bounded) 수 있습니다. 예를 들어 고정된 크기의 데이터 집합처럼 말이죠. 플링크는 언바운드 스트림을 처리하기 위한 세련된 기능 뿐만 아니라 바운드 스트림을 효율적으로 처리하기 위한 전용 연산자 또한 가지고 있습니다.
- **실시간, 저장된 스트림**: 모든 데이터는 스트림으로 생성됩니다. 데이터를 처리하기 위한 방법으로는 두 가지 방식ㅡ생성이 되면 실시간으로 처리하는 방식과 저장소 시스템(ex. 파일 시스템, 객체 저장소)에 스트림을 유지하고 나중에 처리하는 방식ㅡ이 존재합니다. 플링크 어플리케이션은 저장된 스트림과 실시간 스트림을 모두 처리할 수 있습니다.

## 상태

모든 non-trivial 스트리밍 어플리케이션은 상태유지적입니다. 개별적인 이벤트에 변환을 적용하는 어플리케이션만이 상태를 필요로 하지 않죠. 기본적인 비즈니스 로직을 실행하는 모든 어플리케이션은 이벤트나 즉각적인 결과에 추후(다음 이벤트를 수신했을 때나 특정 시간 이후) 액세스하기 위하여 그것들을 기억해둘 필요가 있습니다.

어플리케이션 상태(state)는 플링크의 일등 시민과도 같습니다. 상태 핸들링 컨텍스트에서 플링크가 제공하는 모든 기능을 살펴봄으로써 이를 확인해볼 수 있습니다.

- **다중 상태 원시값** : 플링크는 원자값, 리스트 또는 맵과 같은 다양한 자료 구조에 대해 상태 원시값을 제공합니다. 개발자는 기능의 액세스 패턴을 기반으로 가장 효율적인 상태 원시값을 선택할 수 있습니다.
- **플러그 가능 상태 백엔드** : 어플리케이션 상태는 '플러그 가능 상태 백엔드'에 의해 관리 및 검사됩니다. 플링크는 메모리 또는 효율적인 내장 디스크 데이터 저장소인 RocksDB에 상태를 저장하는 다양한 상태 백엔드를 제공합니다. 사용자 지정(커스텀) 상태 백엔드 또한 플러그될 수 있습니다.
- **단 한번의 상태 일관성** : 플링크의 검사 및 복구 알고리즘은 장애 발생 시 어플리케이션 상태의 일관성을 보장합니다. 따라서 장애는 명료하게 처리되고 어플리케이션의 정확성에는 영향을 미치지 않습니다.
- **특대규모 상태** : 플링크는 비동기적이고 증진적인 검사 알고리즘 덕에 수 테라바이트 규모의 어플리케이션 상태를 유지할 수 있습니다.
- **확장가능한 어플리케이션** : 플링크는 상태를 더 많은, 혹은 더 적은 작업자에게 재분배함으로써 상태유지적 어플리케이션의 확장을 지원합니다.

## 시간

시간은 스트리밍 어플리케이션의 또 다른 핵심 요소입니다. 각 이벤트는 특정한 시점에 생성되기 때문에 대부분의 이벤트 스트림은 내재적인 시간 시맨틱을 가집니다. 또한 윈도우 집계, 세션화, 패턴 감지, 시간 기반 조인 등과 같은 대다수의 일반적인 스트림 연산은 시간을 기반으로 합니다. 스트림 처리의 핵심 사항은 어플리케이션이 시간을 측정하는 방식(ex. 이벤트 시간과 프로세싱 시간의 차이)입니다.

플링크는 다양한 시간 관련 기능을 제공합니다.

- **이벤트 시간 모드**: 이벤트 시간 시맨틱을 갖는 스트림을 처리하는 어플리케이션은 이벤트의 타임스탬프를 기반으로 결과를 계산합니다. 이 덕에 이벤트 시간 프로세싱은 어떤 종류의 이벤트(저장된 이벤트 또는 실시간 이벤트)가 처리되든지에 관계없이 정확하고 일관적인 결과를 산출할 수 있습니다.
- **워터마크 지원**: 플링크는 이벤트 시간 어플리케이션에서 시간을 추론하기 위해 워터마크를 사용합니다. 워터마크는 지연과 결과 완전성의 균형에 유연한 매커니즘이기도 합니다.
- **지연 데이터 핸들링**: 이벤트 시간 모드에서 워터마크가 있는 스트림을 처리할 때는 관련된 모든 이벤트가 도달하기 전에 연산이 완료 되었다고 간주되는 일이 발생할 수 있습니다. 이러한 이벤트를 지연 이벤트라고 합니다. 플링크는 사이드 아웃풋을 통해 재라우팅하거나 완료된 결과를 미리 업데이트하는 등 지연 이벤트를 다루기 위한 여러 옵션을 사용합니다.
- **처리 시간 모드**: 이벤트 시간 모드와 더불어, 플링크는 처리기(processing machine)의 WCT(Wall Clock Time)에 의해 트리거된 연산을 수행하는 처리 시간 시맨틱 또한 지원합니다. 처리 시간 모드는 대략적인 결과 허용이 가능한 엄격한 저지연성을 요구하는 특정 어플리케이션에 적합할 수 있습니다.

## 계층형 API

플링크는 세 개로 계층화된 API를 제공합니다. 각 API는 간결성과 표현성 사이의 각각 다른 균형을 제공하며 각기 다른 유스케이스를 대상으로 합니다.

각 API에 대해 간단히 설명한 후 해당 API들의 어플리케이션에 대해 논의해보고 예시 코드를 살펴보겠습니다.

### 처리함수

처리함수(ProcessFunction)는 플링크가 제공하는 것들 중 가장 표현적인expressive 함수 인터페이스입니다. 플링크는 윈도우로 그룹화된 한 개 또는 두 개의 입력 스트림이나 이벤트로부터 수신한 개별적인 이벤트를 처리하기 위해 처리함수를 제공합니다. 처리함수는 상태 및 시간에 대해 정밀한 제어를 제공합니다. 처리함수는 그것의 상태를 임의로 변경하고 추후 콜백 함수를 트리거할 타이머를 등록할 수 있습니다. 따라서 처리함수는 많은 상태유지적 이벤트-기반 어플리케이션을 위해 요구되는 복잡한 이벤트 단위의 비즈니스 로직을 구현할 수 있습니다.

다음 예제는 `KeyedStream` 상에서 작동하고 START와 END 이벤트를 매칭하는 `KeyedProcessFunction`을 보여줍니다. START 이벤트를 수신하면 해당 함수는 상태 내부에 이벤트의 타임스탬프를 기억해두고 4시간 이후로 타이머를 등록합니다. 타이머가 발현하기 전에 END 이벤트를 수신하면 함수는 END와 START 이벤트 사이의 시간 간격을 계산하고 상태를 해제한 후 값을 반환합니다. 그렇지 않은 경우에는 타이머가 울리고 상태가 해제됩니다.

- Matches keyed START and END events and computes the difference between
- both elements' timestamps. The first String field is the key attribute,
- the second String attribute marks START and END events.

```
public static class StartEndDuration
  extends KeyedProcessFunction<String, Tuple2<String, String>, Tuple2<String, Long>> {

  private ValueState<Long> startTime;

  @Override
  public void open(Configuration conf) {
    // obtain state handle
    startTime = getRuntimeContext()
      .getState(new ValueStateDescriptor<Long>("startTime", Long.class));
  }

  /** Called for each processed event. */
  @Override
  public void processElement(
      Tuple2<String, String> in,
      Context ctx,
      Collector<Tuple2<String, Long>> out) throws Exception {

    switch (in.f1) {
      case "START":
        // set the start time if we receive a start event.
        startTime.update(ctx.timestamp());
        // register a timer in four hours from the start event.
        ctx.timerService()
          .registerEventTimeTimer(ctx.timestamp() + 4 * 60 * 60 * 1000);
        break;
      case "END":
        // emit the duration between start and end event
        Long sTime = startTime.value();
        if (sTime != null) {
          out.collect(Tuple2.of(in.f0, ctx.timestamp() - sTime));
          // clear the state
          startTime.clear();
        }
      default:
        // do nothing
    }
  }

  /** Called when a timer fires. */
  @Override
  public void onTimer(
    long timestamp,
    OnTimerContext ctx,
    Collector<Tuple2<String, Long>> out) {

    // Timeout interval exceeded. Cleaning up the state.
    startTime.clear();
  }
}
```

위 예제는 `KeyedProcessFunction`의 표현성에 대한 강점을 보여주지만 그것이 다소 장황한 인터페이스임을 강조하기도 합니다.

### 데이터스트림 API

데이터스트림 API는 윈도잉, 일괄 레코드 변환, 외부의 데이터 저장소 쿼리를 통한 이벤트 강화 등 많은 일반적인 스트림 처리 연산에 대한 원시값을 제공합니다. 데이터스트림 API는 Java 또는 Scala에서 사용할 수 있고, `map()`, `reduce()`, `aggregate()`와 같은 함수를 기반으로 합니다. 함수는 인터페이스를 확장하거나 Java 또는 Scala 람다 함수로 정의할 수 있습니다.

다음 예제는 클릭 스트림을 세션화하고 세션 당 클릭수를 카운트하는 방법을 보여줍니다.

```
// a stream of website clicks
DataStream<Click> clicks = ...

DataStream<Tuple2<String, Long>> result = clicks
  // project clicks to userId and add a 1 for counting
  .map(
    // define function by implementing the MapFunction interface.
    new MapFunction<Click, Tuple2<String, Long>>() {
      @Override
      public Tuple2<String, Long> map(Click click) {
        return Tuple2.of(click.userId, 1L);
      }
    })
  // key by userId (field 0)
  .keyBy(0)
  // define session window with 30 minute gap
  .window(EventTimeSessionWindows.withGap(Time.minutes(30L)))
  // count clicks per session. Define function as lambda function.
  .reduce((a, b) -> Tuple2.of(a.f0, a.f1 + b.f1));
```

### SQL, TABLE API

플링크에는 두 개의 관계형 API인 Table API와 SQL이 존재합니다. 두 API는 배치와 스트림 프로세싱에 대해 통합된 API입니다. 가령 쿼리는 제한이 없는 실시간 스트림 또는 제한이 있는 저장된 스트림에 대해 동일한 의미로 실행되고 동일한 결과를 산출합니다. Table API와 SQL은 파싱, 검증, 쿼리 최적화를 위해 Apache Calcite를 활용합니다. 두 API는 데이터스트림 및 데이터셋 API와 균일하게 통합될 수 있으며 사용자 정의 스칼라, 집계, 테이블값 함수를 지원합니다.

플링크의 관계형 API는 데이터 분석, 데이터 파이프라인, ETL 어플리케이션을 쉽게 정의할 수 있도록 설계되었습니다.

다음 예제는 클릭 스트림을 세션화하고 세션 당 클릭수를 카운트하는 SQL 쿼리를 보여줍니다. 이 예제는 데이터스트림 API의 예제와 동일한 유스케이스입니다.

```
SELECT userId, COUNT(*)
FROM clicks
GROUP BY SESSION(clicktime, INTERVAL '30' MINUTE), userId
```

## 라이브러리

플링크는 일반적인 프로세싱 유스케이스를 위해 몇가지 라이브러리를 사용합니다. 라이브러리들은 완전히 자립적이지 않고, 전적으로 API에 내장되어 있습니다. 따라서 API의 모든 기능을 활용할 수 있고 다른 라이브러리와 통합될 수 있습니다.
